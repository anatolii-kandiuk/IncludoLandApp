# Звіт по модулю Machine Learning

## 1. Загальна інформація

**Назва проєкту:** IncludoLand  
**Предметна область:** підтримка навчально-корекційного процесу дітей через ігрові активності  
**Модуль:** прогнозування результативності активностей на базі Machine Learning  
**Дата підготовки звіту:** 05.03.2026

Метою ML-модуля є автоматизований аналіз історичних результатів дитини в активностях платформи та формування прогнозу наступного результату, оцінки динаміки прогресу, орієнтовного часу до досягнення майстерності та текстових рекомендацій для спеціаліста.

---

## 2. Цілі та задачі ML-модуля

### 2.1. Цілі
- Підвищити обґрунтованість рішень спеціаліста за рахунок аналітики даних.
- Ранньо виявляти стагнацію або погіршення динаміки навчання.
- Надавати зрозумілі рекомендації українською мовою на основі прогнозу.
- Інтегрувати прогнозний механізм у наявний Django-контур без окремого сервісу.

### 2.2. Функціональні задачі
- Збір і підготовка історичних даних з таблиці `GameResult`.
- Побудова ознак часової динаміки (rolling/statistical/trend features).
- Навчання регресійної моделі (`LinearRegression` або `XGBRegressor`).
- Збереження/завантаження моделі, scaler та метрик.
- Прогноз для конкретного учня і типу активності.
- Розрахунок confidence, прогнозу досягнення порогу майстерності (90+).
- Повернення результату через API та відображення у кабінеті спеціаліста.

## 4. Джерела даних та підготовка

### 4.1. Джерело
Основне джерело — модель `accounts.models.GameResult`.

Ключові поля, що використовуються:
- `user_id`, `game_type`
- `score`
- `duration_seconds`
- `raw_score`, `max_score`
- `details` (словник з похідними метриками)
- `max_streak`
- `created_at`

### 4.2. Нормалізація та валідація
У `extract_game_data(...)` виконуються:
- фільтрація за `user_id`/`game_type`;
- сортування за `(user_id, game_type, created_at)`;
- безпечне приведення значень до невід’ємних цілих;
- реконструкція `successful_attempts` з fallback-логікою;
- обчислення `time_of_day` (кодування періоду доби: ранок/день/вечір-ніч);
- відсікання груп із недостатньою кількістю записів (`min_entries`).

### 4.3. Обробка помилок
При відсутності достатньої кількості записів піднімається `ValueError`, який перехоплюється на рівні сервісу/в’юхи з поверненням керованого повідомлення користувачу.

## 5. Інженерія ознак

### 5.1. Формування train-вибірки
Функція `preprocess_features(df, window_size=3)` генерує ознаки за ковзним вікном попередніх спроб, а цільова змінна (`y`) — `score` наступної спроби.

### 5.2. Набір ознак моделі
У `ProgressPredictor.FEATURE_COLUMNS` використано 14 ознак:
1. `attempt_number`
2. `avg_score`
3. `std_score`
4. `avg_duration`
5. `total_hints`
6. `avg_successful_attempts`
7. `avg_failed_attempts`
8. `failed_attempts_trend`
9. `avg_max_streak`
10. `time_of_day`
11. `score_trend`
12. `last_score`
13. `score_improvement`
14. `days_since_start`

### 5.3. Особливості обчислення
- `score_trend` обчислюється як кутовий коефіцієнт лінійної апроксимації (`np.polyfit`).
- `score_improvement` — різниця між двома останніми спробами у вікні.
- `days_since_start` — часовий відступ від першої спроби в групі.
- Пропуски заповнюються нулями (`fillna(0)`).

## 6. Моделі та навчання

### 6.1. Доступні моделі
- `linear` → `LinearRegression`.
- `xgboost` → `XGBRegressor` з параметрами:
  - `n_estimators=300`
  - `learning_rate=0.05`
  - `max_depth=10`
  - `subsample=0.9`
  - `colsample_bytree=0.9`
  - `objective='reg:squarederror'`
  - `random_state=42`
  - `n_jobs=-1`

### 6.2. Процес навчання
Метод `train(...)` виконує:
1. Витяг даних і генерацію ознак.
2. `train_test_split` (`test_size=0.2`, `random_state=42`).
3. Масштабування через `StandardScaler`.
4. Навчання моделі.
5. Оцінку якості на train/test.

### 6.3. Метрики якості
Розраховуються:
- `MAE`
- `RMSE`
- `R²`

Додатково зберігаються:
- `n_samples`
- `n_features`

### 6.4. Збереження артефактів
Після навчання зберігаються:
- модель: `progress_predictor_<model_type>_<game_suffix>.joblib`
- scaler: `scaler_<model_type>_<game_suffix>.joblib`
- метрики: `metrics_<model_type>_<game_suffix>.json`

Каталог артефактів: `ml_models/`.

## 7. Логіка прогнозування

### 7.1. Вхідні параметри
Для прогнозу потрібні:
- `user_id`
- `game_type`

### 7.2. Алгоритм
1. Перевірка, що модель натренована/завантажена.
2. Отримання user-level ознак через `extract_user_features(...)`.
3. Масштабування ознак через наявний `StandardScaler`.
4. Обчислення `predicted_score`.
5. Кліпінг прогнозу до діапазону `[0, 100]`.
6. Обчислення confidence:
  - для `LinearRegression` та `XGBoost` — через `test_rmse`.
7. Генерація текстового інсайту (`_generate_insight`).
8. Оцінка часу/спроб до майстерності (`_estimate_mastery`).

### 7.3. Бізнес-правила
- Якщо `last_score >= 90` і позитивний тренд, застосовується обмеження від «різкого провалу» прогнозу.
- Якщо `score_trend <= 0.1`, оцінка до майстерності не дається (`None`).
- Для вже досягнутої майстерності (`current_score >= 90`) повертаються `(0, 0)`.

---

## 8. Евристичний fallback (без повноцінного ML)

У `predict_performance` реалізовано резервний режим `analysis_mode='heuristic'` для випадків:
- модель відсутня;
- недостатньо даних для навчання;
- історія активності коротка (менше 10 записів у вибірці для UI).

Евристичний прогноз базується на:
- середньому балу в останньому вікні,
- локальному тренді,
- обмеженні в діапазоні 0..100,
- базовому confidence через стандартне відхилення.

Це дозволяє повертати корисний аналітичний результат навіть на ранніх етапах накопичення даних.

## 11. Результати тестування та перевірки

### 11.1. Наявні перевірки
Файл `test_ml_insights.py` призначений для ручної перевірки генерації текстових інсайтів українською мовою за наборами сценаріїв.

### 11.2. Виявлена невідповідність
У тесті виклик `_generate_insight(...)` передає параметр `avg_score`, тоді як фактична сигнатура методу в `ProgressPredictor` цього параметра не містить.

Очікуваний технічний наслідок: `TypeError` при запуску тесту без адаптації.

Рекомендація:
- або прибрати `avg_score` з виклику в тесті;
- або додати опційний параметр у сигнатуру методу (якщо потрібен у логіці).

## 13. Обмеження та ризики

1. **Нестабільність hash-кодування `game_type`:** використання `hash()` може відрізнятися між процесами/сесіями Python.
2. **Обмежена формальна тестова база:** є ручний скрипт, але не повний набір автоматизованих unit/integration тестів для всього ML pipeline.
3. **Порожній каталог `ml_models/` у поточному стані workspace:** перед продакшн-використанням потрібне первинне навчання і збереження артефактів.
4. **Якість прогнозу критично залежить від обсягу і чистоти історичних даних.**
5. **Вибірка game types у UI відрізняється від повного словника типів:** окремі типи можуть бути виключені на рівні сторінки спеціаліста.

---

## 14. Пропозиції щодо покращення

1. Замінити нестабільний `hash(game_type)` на детерміноване кодування (наприклад, фіксований словник/OneHotEncoder).
2. Додати набір unit-тестів:
   - extraction/preprocessing,
   - prediction/fallback,
   - API status/error contracts.
3. Реалізувати періодичне перевчання моделі (cron/Celery beat) та контроль дрейфу метрик.
4. Додати версіонування артефактів моделей і журнал змін моделей.
5. Розширити explainability-блок (важливість ознак, пояснення причин прогнозу для спеціаліста).

## 15. Висновок

ML-модуль у проєкті IncludoLand реалізовано як прикладний, інтегрований у Django механізм прогнозування прогресу учня за даними ігрової активності. Рішення охоплює повний цикл: від підготовки даних і навчання моделі до API-видачі прогнозу та його візуалізації у кабінеті спеціаліста.

Наявність fallback-режиму забезпечує стабільну поведінку навіть при дефіциті даних, а генерація текстових інсайтів робить результати доступними для практичного педагогічного застосування. Поточна реалізація придатна для експлуатації в межах навчального проєкту та має чіткий потенціал до подальшого розвитку в напрямі тестованості, детермінованості ознак і MLOps-практик.
